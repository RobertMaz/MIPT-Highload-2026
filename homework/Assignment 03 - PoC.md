## Ваша VM

Каждая команда получает выделенную VM:

| Ресурс | Объём     |
|--------|-----------|
| CPU    | 2 vCPU    |
| RAM    | 8 GB      |
| Disk   | 72 GB HDD |

Всё приложение (сервисы, БД, очереди, кэш) крутится на VM через `docker compose`. Ресурсов мало намеренно — "кинуть железа" нельзя, придётся
оптимизировать.

Нагрузочное тестирование запускайте с вашего ноутбука (или другой машины), а не с VM — чтобы load generator не конкурировал за ресурсы с
приложением.

## Задание

### Часть 1: Реализация PoC

Реализуйте ядро вашей системы — минимальный набор компонентов, достаточный для проверки архитектуры.

Минимум:

- 1–3 микросервиса с сетевым взаимодействием (БД/очередь по сети, не in-process)
- 2–3 API-операции из Задания 2 (основной happy path)
- База данных (реальная, не in-memory mock)

Nice to have (бонус при оценке): асинхронное взаимодействие через очередь сообщений — RabbitMQ, Kafka, Redis Streams.

Технологии: любой язык, любой фреймворк.

Потолок: 1–3 сервиса + БД + очередь + опционально cache/nginx. Больше — обоснование в README.

Prometheus/Grafana на VM не запускайте — съедят ресурсы, которые нужны вашему приложению. Метрики смотрите через `docker stats`, `htop`,
`iostat` на VM + вывод load-тестера (k6/Locust дают свои дашборды и отчёты на стороне клиента).

Resource limits (обязательно): в docker-compose для каждого контейнера. Без лимитов один контейнер съест всю RAM и убьёт остальные.

```yaml
services:
  api:
    deploy:
      resources:
        limits:
          cpus: "0.5"
          memory: 512M
```

Docker Compose для запуска всего стека одной командой:

```bash
docker compose up
```

Работает из коробки (обязательно). `git clone → docker compose up → healthcheck зелёный → curl API → 200 OK`. Если PoC не поднимается у
проверяющего с первой попытки — 0 баллов за Часть 1. Никаких "допишите в hosts", "закомментируйте строку 42 в docker-compose".

### Часть 2: Паттерны

Примените минимум 4 паттерна: 2+ проектирования и 2+ устойчивости. Какие именно — решаете вы, исходя из архитектуры.

Примеры (не обязательный список):

- *Проектирование:* API Gateway, CQRS, Saga, Outbox, Event Sourcing, Cache-Aside, BFF
- *Устойчивость:* Circuit Breaker, Retry with backoff, Timeout, Bulkhead, Rate Limiting, Fallback, Health Check

Для каждого паттерна в README:

- Какой и где применили
- Какую проблему решает
- Ссылка на код

### Часть 3: Итеративная оптимизация

Главная цель: выжать из VM максимум RPS при сбалансированной загрузке ресурсов.

- ✅ Хорошо: высокий RPS при CPU 70–90% + RAM используется + disk I/O под нагрузкой. Сервер работает на полную.
- ❌ Плохо: 20 RPS при CPU 100% — код неэффективен (N+1, блокировки, отсутствие пула), сервер задыхается не от нагрузки а от себя.
- ❌ Плохо: 300 RPS при CPU 20% — bottleneck в другом месте (БД, connection pool, сеть, threadpool), а CPU простаивает.

Хорошая оптимизация = ресурсы загружены равномерно до предела, а не выборочно.

Цикл: замер → анализ bottleneck → оптимизация → замер.

Инструменты: k6, Locust, Gatling, wrk, jmeter — на ваш выбор.

Где искать bottleneck (наводки — дальше думайте сами):

- Slow queries: `EXPLAIN ANALYZE`, `pg_stat_statements`, slow query log — встроено в БД, ничего ставить не нужно.
- Хотите Prometheus/Grafana — поднимите на своей машине, scrape'айте экспортёры с VM по сети (node-exporter, cAdvisor, app `/metrics`). Метрики собираются вне VM и не едят её ресурсы.

Чекпоинт (24.04): к этой дате PoC должен запускаться на VM, а Iteration 0 (baseline) — зафиксирован в `docs/optimization-log.md`. Это не
отдельная оценка, но без чекпоинта не останется времени на итерации — а они стоят 6 из 10 баллов.

#### Целевые метрики

Достигните NFR из Задания 1. Минимальная планка (если NFR из ДЗ 1 мягче — всё равно доберитесь до этой):

| Метрика         | Требование                                                                                        |
|-----------------|---------------------------------------------------------------------------------------------------|
| Read-операции   | ≥ 100 RPS при p99 < 500ms                                                                         |
| Write-операции  | ≥ 30 RPS при p99 < 1s                                                                             |
| Error rate      | < 1% при устойчивой нагрузке (минимум 5 минут на целевом RPS)                                     |
| 2x spike        | error rate < 5%, нет OOM / крашей контейнеров, система восстанавливается за 1 минуту после спайка |
| Баланс ресурсов | CPU 70–90% ИЛИ RAM 70–90% ИЛИ disk I/O saturated на пике RPS                                      |

Профиль трафика (read-heavy / write-heavy / mixed) зафиксируйте в README. Если read-heavy — write-планка может быть снижена до 10 RPS, и
наоборот. Полностью обнулить одну из планок нельзя.

Если не достигли своих NFR из ДЗ 1 — это нормальный результат. Зафиксируйте gap и объясните: что bottleneck, что нужно изменить (ресурсы,
архитектуру, сами NFR), сколько это будет стоить.

#### Итерация 0: Baseline

Запустите PoC на VM, сначала прогрейте систему (1–2 минуты на низком RPS — JVM JIT, кэши БД, пулы соединений), потом проведите stress-тест (
плавно увеличивайте RPS до отказа).

Длительность устойчивой нагрузки на каждой ступени — минимум 2 минуты. p99 за 30 секунд — не p99.

Зафиксируйте метрики USE + RED (из лекции 17.04):

RED (сервис — что видит клиент):

- Rate: max RPS до деградации
- Errors: error rate (%)
- Duration: p50, p95, p99 latency

USE (ресурсы — что происходит внутри):

- Utilization: CPU, RAM (`docker stats`, `htop`)
- Saturation: Disk I/O, connection queue (`iostat`)
- Errors: OOM, connection refused

Анализ:

- Bottleneck: что именно стало узким местом?
- Gap vs NFR: на сколько baseline отстаёт от ваших NFR из Задания 1?

#### Итерации 1–3: Оптимизация

Каждая итерация: найти bottleneck → исправить → замерить → показать разницу.

Порядок определяете вы — зависит от того, что показал baseline. Типичные направления:

- Код: N+1 запросы, отсутствие connection pooling, синхронные вызовы, утечки соединений
- Данные: индексы (`EXPLAIN ANALYZE` до/после), кэширование (Redis), тюнинг запросов, connection pool
- Масштабирование: несколько инстансов сервиса за nginx, read replica

На каждую итерацию в `docs/optimization-log.md` — 1–2 абзаца (что нашли, что сделали, что изменилось) + строка в таблице прогресса.

## Критерии оценки

Шкала на критерий: 0 — не выполнено · 1 — выполнено частично · 2 — выполнено полностью (для 4-балльной строки: 0/2/4).

| Критерий                                                                                                | Макс |
|---------------------------------------------------------------------------------------------------------|------|
| PoC: `docker compose up` → API отвечает 200 (end-to-end через все сервисы), resource limits проставлены | 2    |
| Паттерны: 2+ design + 2+ resilience с обоснованием (что / зачем / ссылка на код)                        | 2    |
| Итерация 0: baseline с USE+RED метриками + bottleneck analysis (что узкое, почему)                      | 2    |
| 2+ итерации оптимизации: bottleneck → fix → замер → сравнение (цифры до/после в таблице)                | 4    |

Максимум: 10 баллов

Критерии частичного зачёта (1 балл):

- PoC поднимается, но часть сервисов работает некорректно / нет resource limits
- Паттерны есть, но без ссылок на код или без обоснования "зачем"
- Baseline есть, но нет USE (только RED) или нет bottleneck analysis
- 1 итерация оптимизации вместо 2 (= 2 из 4 баллов)
