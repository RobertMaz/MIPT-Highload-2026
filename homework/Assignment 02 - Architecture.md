## Задание

### Часть 1: Архитектура системы (5–10 компонентов)

Спроектируйте архитектуру. Компонент = всё, что едет в прод как отдельная единица (сервис, БД, очередь, кэш, search, CDN). 5–10 в сумме.

Архитектурный стиль:
Зафиксируйте выбранный стиль и обоснуйте выбор — почему он подходит под ваши требования и ограничения. Варианты: Layered (слоистая),
Microservices (MSA), Event-Driven (EDA), Service-Based (SBA) или гибрид. Это может быть одним из ваших ADR (Часть 5).

C4 диаграммы:

1. Level 1 — System Context: ваша система, пользователи и внешние системы
2. Level 2 — Container Diagram: основные контейнеры (сервисы, БД, очереди, кэши)

Для каждого компонента укажите:

- Назначение (1–2 предложения)
- Технология (язык, фреймворк)
- Тип коммуникации с другими компонентами (sync/async)

### Часть 2: Sequence Diagrams

Нарисуйте sequence diagrams для 3 ключевых сценариев:

1. Happy path — основной пользовательский сценарий
2. Сценарий с ошибкой — отказ одного из компонентов (таймаут, недоступность)
3. Асинхронный сценарий — обработка в фоне, уведомление, saga

### Часть 3: API Design

Спроектируйте REST или gRPC API для 3–5 основных операций.

Для каждого endpoint укажите: метод + путь, request/response JSON, коды ошибок, подход к версионированию.

Пример формата (адаптируйте под свой кейс):

```
POST /api/v1/orders
Описание: Создание нового заказа
Request:
  {
    "restaurant_id": "uuid",
    "items": [{"menu_item_id": "uuid", "quantity": 1}],
    "delivery_address": {"lat": 55.75, "lon": 37.62}
  }
Response 201:
  {
    "order_id": "uuid",
    "status": "created",
    "estimated_delivery": "2026-05-01T18:30:00Z"
  }
Errors:
  400 — невалидные данные
  404 — ресторан не найден
  429 — rate limit exceeded
  503 — сервис временно недоступен
```

Можно использовать OpenAPI: https://habr.com/ru/companies/first/articles/973186/

### Часть 4: Выбор БД и модель данных

Выбор БД:

- Для каждого хранилища укажите тип (PostgreSQL, MongoDB, Redis, ClickHouse, Elasticsearch, etc.)
- Обоснуйте выбор: почему именно этот тип под ваш паттерн доступа

Модель данных — 3–5 основных сущностей (таблиц/коллекций) в сумме по всем БД:

- Ключи (PK, FK / partition key)
- Индексы с обоснованием
- Примерный объём записей (из capacity estimation в Задании 1)

### Часть 5: ADR (Architecture Decision Records)

Напишите 2–3 ADR для ключевых архитектурных решений.

## Критерии оценки

Шкала на критерий: 0 — не выполнено · 1 — выполнено частично · 2 — выполнено полностью.

| Критерий                                                                              | Макс |
|---------------------------------------------------------------------------------------|------|
| Архитектурный стиль обоснован (почему именно он), C4 L1 + L2 с описаниями компонентов | 2    |
| Sequence diagrams: happy path + ошибка + async (3 отдельные диаграммы)                | 2    |
| API: 3–5 операций с request/response/errors + подход к версионированию                | 2    |
| Выбор БД обоснован (под паттерн доступа), модель данных: PK/FK, индексы, объём        | 2    |
| ADR: 2–3 решения, у каждого — контекст, альтернативы (2+), последствия                | 2    |

Максимум: 10 баллов

Критерии частичного зачёта (1 балл):

- Стиль назван без обоснования / C4 без описаний компонентов
- Sequence: happy path есть, но ошибка или async пропущены / одна диаграмма на всё
- API: endpoint'ы без кодов ошибок или без версионирования
- БД выбрана без обоснования / модель данных без индексов
- ADR без альтернатив (например, "выбрали PostgreSQL, потому что подходит")
